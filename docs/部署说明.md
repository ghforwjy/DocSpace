## 项目架构概述

DocSpace 是基于微服务架构的协作平台：

| 类别 | 技术栈 | 微服务列表 |
|------|--------|-----------|
| **后端服务** | .NET | API、Studio、Files、Files.Service、People、Backup、Backup.BackgroundTasks、Notify、Studio.Notify、ClearEvents、HealthChecks、Telegram、AI、AI.Service、ApiSystem |
| **身份服务** | Java (Spring) | Identity Authorization、Identity API |
| **前端服务** | Node.js | Login、SDK、DocEditor、Management、Socket.IO、SSOAuth |
| **路由服务** | Nginx | Router |
| **基础设施** | - | MySQL、Redis、RabbitMQ、OpenSearch、Document Server |

### 基础设施依赖

| 服务 | 用途 | 默认端口 |
|------|------|---------|
| MySQL 8.0 | 主数据库 | 3306 |
| Redis 7.0 | 缓存服务 | 6379 |
| RabbitMQ 3.11 | 消息队列 | 5672 / 15672 |
| OpenSearch | 日志存储 | 9200 / 9600 |
| Document Server | 文档编辑 | 80 |

## 镜像构建

### 构建配置文件

| 文件 | 构建内容 |
|------|----------|
| `build.yml` | 所有后端 .NET 微服务镜像（每个微服务独立镜像） |
| `build-identity.yml` | Identity Java 微服务镜像 |
| `client/docker/Dockerfile` | 前端所有 Node.js 微服务（聚合为一个镜像） |

### 构建命令

```bash
cd buildtools/install/docker

# 构建后端所有 .NET 微服务镜像
docker compose -f build.yml build

# 构建 Identity Java 微服务
docker compose -f build-identity.yml build

# 前端镜像（聚合所有 Node.js 微服务）
cd ../../../client/docker
docker compose build
```

## 部署模式总览

DocSpace 有 **2 种架构模式**，每种模式都有生产部署和开发部署变体：

| 架构模式 | 配置文件 | 微服务容器化方式 | 容器数量 |
|----------|----------|-----------------|----------|
| **聚合模式** | `docspace-stack.yml` | 相邻技术栈聚合：dotnet/java/node 各一个 Supervisor 容器 | 4 个业务容器 |
| **独立模式** | `docspace.yml` + `docspace.profiles.yml` | 每个微服务独立容器 | 25+ 个容器 |

### 架构对比图

```
聚合模式 (docspace-stack.yml)：
┌─────────────────────────────────────────────────────────┐
│  onlyoffice-dotnet-services  │  .NET 所有微服务        │
│  (Supervisor)                │  API/Studio/Files/...  │
├─────────────────────────────────────────────────────────┤
│  onlyoffice-java-services    │  Java 微服务            │
│  (Supervisor)                │  Identity/Authorization │
├─────────────────────────────────────────────────────────┤
│  onlyoffice-node-services   │  Node.js 所有微服务     │
│  (Supervisor)                │  Login/SDK/DocEditor/.. │
├─────────────────────────────────────────────────────────┤
│  onlyoffice-router          │  Nginx 路由             │
└─────────────────────────────────────────────────────────┘

⚠️ **注意**：Document Server 为独立外部服务，不在聚合镜像内，
   需要通过 `ds.yml` 单独启动。

独立模式 (docspace.yml + profiles)：
┌─────────────┐  ┌─────────────┐  ┌─────────────┐
│ onlyoffice  │  │ onlyoffice  │  │ onlyoffice  │
│ -api        │  │ -studio     │  │ -files      │
├─────────────┤  ├─────────────┤  ├─────────────┤
│ onlyoffice  │  │ onlyoffice  │  │ onlyoffice  │
│ -people     │  │ -notify     │  │ -backup     │
├─────────────┤  ├─────────────┤  ├─────────────┤
│ onlyoffice  │  │ onlyoffice  │  │ onlyoffice  │
│ -login      │  │ -doceditor  │  │ -sdk        │
└─────────────┘  └─────────────┘  └─────────────┘
  (每个微服务独立容器)
```

### 模式选择建议

| 场景 | 推荐架构 |
|------|----------|
| 生产环境（资源优先） | 聚合模式 |
| 开发/测试（服务多、调试频繁） | 独立模式 |

## 生产环境部署

### 前提条件

- Docker 20.10+
- Docker Compose 2.0+
- 至少 8GB RAM
- 至少 50GB 磁盘空间

### 启动步骤

#### 1. 启动基础设施

```bash
cd buildtools/install/docker

# 启动 MySQL、Redis、RabbitMQ
docker compose -f db.yml -f redis.yml -f rabbitmq.yml up -d
```

#### 2. 运行数据库迁移

```bash
docker compose -f migration-runner.yml run --rm onlyoffice-migration-runner
```

#### 3. 启动业务服务（二选一）

**聚合模式（生产推荐）**：

```bash
# 启动 Document Server（独立外部服务）
docker compose -f ds.yml up -d

# 启动聚合模式服务
docker compose -f docspace-stack.yml up -d
```

> ⚠️ **Document Server 说明**：聚合模式不包含 Document Server，需要单独启动。

**独立模式（开发/测试推荐）**：

```bash
# 启动所有服务
docker compose -f docspace.yml --profile prod up -d

# 或选择性启动：
# 仅后端微服务
docker compose -f docspace.yml --profile backend up -d

# 仅前端微服务
docker compose -f docspace.yml --profile frontend up -d

# 仅额外基础设施
docker compose -f docspace.yml --profile extra-services up -d
```

## 开发环境部署

开发环境同样支持两种架构模式。

### 后端开发

#### 聚合模式开发

```bash
# 1. 启动基础设施
docker compose -f db.yml -f redis.yml -f rabbitmq.yml up -d

# 2. 启动 Document Server（独立外部服务）
docker compose -f ds.yml up -d

# 3. 运行数据库迁移
docker compose -f migration-runner.yml run --rm onlyoffice-migration-runner

# 4. 启动聚合模式（与生产相同）
docker compose -f docspace-stack.yml up -d
```

**特点**：需要修改代码 → 重新构建镜像 → 重启容器

#### 聚合模式开发 - 二进制挂载（自制）

**注意**：本配置为自制，非 DocSpace 原项目提供。

DocSpace 原项目只为独立模式提供了二进制挂载配置（overcome.yml），未为聚合模式提供。

如果需要在聚合模式下也使用二进制挂载，可以使用 `docspace-stack-overcome.yml`（位于 `buildtools/install/docker/` 目录）。

**核心原理**：

1. **覆盖 Supervisor 配置**：通过 volume 将 `dotnet_services-overcome.conf` 挂载到 `/etc/supervisor/conf.d/supervisord.conf`，覆盖原配置
2. **挂载源码目录**：将宿主机 `${ROOT_DIR}/server` 挂载到容器内 `/var/www/server`
3. **环境变量 BUILD_PATH**：设置为 `/var/www/server`，Supervisor 的 `directory` 已指向 `${BUILD_PATH}/...`

**关键文件**：

| 文件 | 用途 |
|------|------|
| `docspace-stack-overcome.yml` | Docker Compose 配置 |
| `config/supervisor/dotnet_services-overcome.conf` | 自定义 Supervisor 配置 |

**使用方式**：

```bash
# 1. 启动基础设施
docker compose -f db.yml -f redis.yml -f rabbitmq.yml up -d

# 2. 启动 Document Server（独立外部服务）
docker compose -f ds.yml up -d

# 3. 运行数据库迁移
docker compose -f migration-runner.yml run --rm onlyoffice-migration-runner

# 4. 启动聚合模式二进制挂载（需要先设置环境变量 ROOT_DIR）
docker compose -f docspace-stack-overcome.yml up -d
```

**挂载映射**：

| 宿主机 | 容器内 | 说明 |
|--------|--------|------|
| `${ROOT_DIR}/server` | `/var/www/server` | .NET/Node.js 源码（全部） |
| `${ROOT_DIR}/client/public` | `/var/www/public` | 前端静态文件 |
| `${ROOT_DIR}/buildtools/install/docker/config/supervisor/dotnet_services-overcome.conf` | `/etc/supervisor/conf.d/supervisord.conf` | Supervisor 配置覆盖 |
| `${ROOT_DIR}/buildtools/config/appsettings.json` | `/app/onlyoffice/config/appsettings.json` | 配置文件 |

**服务对应目录（容器内 BUILD_PATH=/var/www/server）**：

| 服务 | 容器内路径 |
|------|-----------|
| ASC.AI | `/var/www/server/products/ASC.AI/server` |
| ASC.Files | `/var/www/server/products/ASC.Files/server` |
| ASC.People | `/var/www/server/products/ASC.People/server` |
| ASC.Web.Api | `/var/www/server/services/ASC.Web.Api/service` |
| ASC.Web.Studio | `/var/www/server/services/ASC.Web.Studio/service` |
| ASC.Socket.IO | `/var/www/server/common/ASC.Socket.IO/` |
| ASC.SsoAuth | `/var/www/server/common/ASC.SsoAuth/` |

**特点**：与 docspace-stack.yml 架构相同（4 个聚合容器），但通过 Supervisor 配置覆盖实现源码挂载

#### 独立模式开发 - 二进制挂载

```bash
# 1. 启动基础设施
docker compose -f db.yml -f redis.yml -f rabbitmq.yml up -d

# 2. 运行数据库迁移
docker compose -f migration-runner.yml run --rm onlyoffice-migration-runner

# 3. 启动二进制挂载模式
# 注意：需要先在本地构建二进制到 ${BIN_DIR} 目录
docker compose -f docspace.overcome.yml --profile backend-local up -d
```

**overcome.yml 模式说明**：
- **架构**：独立模式（与 docspace.yml + profiles 相同）
- **容器数量**：约 20 个（每个微服务独立容器）
- **镜像**：使用 `Baseimage_Dotnet_Run` 基础镜像
- **差异**：二进制挂载到本地 BIN_DIR，而非镜像内包含

**二进制挂载映射**：

| 宿主机（BIN_DIR） | 容器内（BUILD_PATH） | 说明 |
|------------------|---------------------|------|
| `ASC.Web.Api/bin/` | `/var/www/studio/ASC.Web.Api/` | API 二进制 |
| `ASC.Web.Studio/bin/` | `/var/www/studio/ASC.Web.Studio/` | Studio 二进制 |
| `ASC.Files/Server/bin/` | `/var/www/products/ASC.Files/server/` | Files 二进制 |
| `ASC.People/Server/bin/` | `/var/www/products/ASC.People/server/` | People 二进制 |
| `ASC.Socket.IO/` | `/var/www/services/ASC.Socket.IO/` | Socket.IO 源码 |
| `ASC.SsoAuth/` | `/var/www/services/ASC.SsoAuth/` | SSOAuth 源码 |
| `client/public/` | `/var/www/public/` | 前端静态文件 |

**独立模式各容器挂载详情**：

| 服务 | 容器内工作目录 | 挂载宿主机路径 |
|------|---------------|---------------|
| onlyoffice-api | `/var/www/studio/ASC.Web.Api/` | `${ROOT_DIR}/server/web/ASC.Web.Api/${BIN_DIR}` |
| onlyoffice-studio | `/var/www/studio/ASC.Web.Studio/` | `${ROOT_DIR}/server/web/ASC.Web.Studio/${BIN_DIR}` |
| onlyoffice-files | `/var/www/products/ASC.Files/server/` | `${ROOT_DIR}/server/products/ASC.Files/Server/${BIN_DIR}` |
| onlyoffice-people | `/var/www/products/ASC.People/server/` | `${ROOT_DIR}/server/products/ASC.People/Server/${BIN_DIR}` |
| onlyoffice-socket | `/var/www/services/ASC.Socket.IO/` | `${ROOT_DIR}/server/common/ASC.Socket.IO/` |
| onlyoffice-ssoauth | `/var/www/services/ASC.SsoAuth/` | `${ROOT_DIR}/server/common/ASC.SsoAuth/` |

**特点**：
- 修改 .NET 代码 → 重新 `dotnet build` → 重启容器（无需重建镜像）
- Node.js 服务挂载源码目录
- 适合频繁修改代码的开发场景

### 前端开发

前端所有微服务聚合为一个镜像，位于 `client/docker/`：

```bash
cd client/docker

# 构建前端镜像（包含所有 Node.js 微服务）
docker compose build

# 启动前端服务
docker compose up -d
```

**暴露端口**：

| 端口 | 微服务 |
|------|--------|
| 5001 | Client 主应用 |
| 5011 | Login |
| 5013 | DocEditor |
| 5015 | Management |
| 5099 | SDK |

**前端开发特点**：
- 使用 `pnpm start-prod` 启动所有 Node.js 微服务
- 修改代码需要重建镜像或使用源码挂载
- 各微服务共享同一个容器

### 完整开发环境组合

```bash
# 终端 1：后端 + 基础设施
cd buildtools/install/docker

# 启动基础设施
docker compose -f db.yml -f redis.yml -f rabbitmq.yml up -d

# 启动后端（选择一种模式）
docker compose -f docspace.overcome.yml --profile backend-local up -d
# 或
docker compose -f docspace-stack.yml up -d

# 终端 2：前端
cd client/docker
docker compose up -d
```

| `RABBIT_HOST` | onlyoffice-rabbitmq | RabbitMQ 主机 |
| `RABBIT_PORT` | 5672 | RabbitMQ 端口 |
| `RABBIT_USER_NAME` | guest | RabbitMQ 用户 |
| `RABBIT_PASSWORD` | guest | RabbitMQ 密码 |
| `APP_CORE_MACHINEKEY` | - | 应用机器密钥（需自定义） |

### 开发模式额外变量

| 变量名 | 说明 |
|--------|------|
| `BIN_DIR` | 本地编译输出目录（如 `bin/Release/net8.0/linux-x64/publish`） |
| `ROOT_DIR` | DocSpace 项目根目录 |
| `BUILD_PATH` | 容器内构建路径（默认 `/var/www`） |

# Linux/macOS

## 端口映射

### 概念说明

| 端口类型 | 说明 |
|---------|------|
| **内部服务端口** | 只在Docker内部网络暴露，用于容器间通信，外部无法直接访问 |
| **映射到宿主机的端口** | 通过Docker ports映射到宿主机，外部可以访问 |
| **暴露给外部的端口** | 直接暴露给外部网络，外部可以直接访问 |

### 聚合模式服务端口（WSL部署使用）

#### 业务服务端口

| 服务 | 容器监听端口 | 端口类型 | 访问地址 |
|------|-------------|---------|---------|
| Router | 80, 8081, 8092, 8099 | 映射到宿主机 | http://localhost:8092（主入口） |
| dotnet-services | 5050 | 内部服务端口 | - |
| java-services | 5050 | 内部服务端口 | - |
| node-services | 5050 | 内部服务端口 | - |

#### 文档编辑服务端口

| 服务 | 容器监听端口 | 端口类型 | 访问地址 |
|------|-------------|---------|---------|
| Document Server | 80 | 映射到宿主机 | http://localhost:8085（内部调用） |

#### 基础设施服务端口

| 服务 | 容器监听端口 | 端口类型 | 访问地址 |
|------|-------------|---------|---------|
| MySQL | 3306 | 映射到宿主机 | localhost:3306 |
| Redis | 6379 | 映射到宿主机 | localhost:6379 |
| RabbitMQ | 5672, 15672 | 映射到宿主机 | localhost:15672（管理界面） |

**用户只需访问 http://localhost:8092 即可使用DocSpace全部功能。**

## 常见问题

### Q1: 启动顺序是什么？

1. 基础设施服务（MySQL、Redis、RabbitMQ）
2. 数据库迁移（migration-runner）
3. 后端服务
4. 前端服务

### Q2: 如何选择部署模式？

| 场景 | 推荐模式 |
|------|----------|
| 生产环境（资源优先） | 聚合模式 docspace-stack.yml |
| 开发/测试（调试优先） | 独立模式 docspace.yml |
| 后端代码频繁修改 | overcome 模式（二进制挂载） |
| 前端开发 | client/docker/docker-compose.yml |

### Q3: overcome 模式和普通独立模式的区别？

| 特性 | docspace.yml 独立模式 | overcome.yml 二进制挂载 |
|------|----------------------|-------------------------|
| 架构 | 独立模式 | 独立模式 |
| 容器数量 | 25+ 个 | 约 20 个 |
| 二进制来源 | 镜像内包含 | 挂载本地 BIN_DIR |
| 镜像构建 | 需要 | 需要 |
| 代码修改生效 | 需重建镜像 | 需重编译 + 重启容器 |
| 重建镜像频率 | 高 | 低 |
| 适用场景 | 生产/偶尔修改 | 频繁修改调试 |

### Q4: 如何查看服务日志？

```bash
# 查看所有日志
docker compose logs -f

# 查看特定微服务日志
docker compose logs -f onlyoffice-api

# 查看最近 100 行日志
docker compose logs --tail 100 onlyoffice-studio
```

### Q5: 如何清理所有容器和数据？

```bash
# 停止所有容器
docker compose down

# 删除容器和网络
docker compose down -v

# 删除所有数据和卷（慎用）
docker compose down -v --remove-orphans
```

### Q6: 镜像构建失败怎么办？

```bash
# 清理 Docker 构建缓存
docker builder prune -a

# 重新构建
docker compose -f build.yml build --no-cache
```

## 附件：WSL部署新增/修改文件清单

本文档记录在WSL Ubuntu环境中部署DocSpace聚合模式时，在原项目基础上新增或修改的配置文件。

### 新增配置文件

| 文件路径 | 用途 | 说明 |
|---------|------|------|
| `buildtools/install/docker/docspace-ports.yml` | 聚合模式端口映射 | 将Docker内部端口映射到Windows宿主机，使Windows可访问服务 |
| `buildtools/install/docker/ds-ports.yml` | Document Server端口映射 | 将Document Server的80端口映射到8085 |

### 创建的配置（不在项目目录）

| 文件路径 | 用途 | 说明 |
|---------|------|------|
| `daemon.json` | Docker镜像加速配置 | 配置国内镜像加速器（ustc、163），位于 `/etc/docker/daemon.json` |

### 配置文件说明

#### docspace-ports.yml
```yaml
services:
  onlyoffice-router:
    ports:
      - "8081:8081"
      - "8092:8092"
      - "8099:8099"

  onlyoffice-dotnet-services:
    ports:
      - "8050:5050"

  onlyoffice-java-services:
    ports:
      - "8051:5050"

  onlyoffice-node-services:
    ports:
      - "8052:5050"
```

#### ds-ports.yml
```yaml
services:
  onlyoffice-document-server:
    ports:
      - "8085:80"
```

### 使用方式

```powershell
# 启动聚合模式（带端口映射）
docker compose -f docspace-stack.yml -f docspace-ports.yml up -d

# 启动Document Server（带端口映射）
docker compose -f ds.yml -f ds-ports.yml up -d
```

### 端口映射详情

| 服务 | 容器端口 | 宿主机端口 | 访问地址 | 用途 |
|------|---------|-----------|---------|------|
| Router | 8092 | 8092 | http://localhost:8092 | 用户主入口 |
| Router | 8081 | 8081 | http://localhost:8081 | API入口 |
| Router | 8099 | 8099 | http://localhost:8099 | 备用入口 |
| dotnet-services | 5050 | 8050 | http://localhost:8050 | 内部服务 |
| java-services | 5050 | 8051 | http://localhost:8051 | 内部服务 |
| node-services | 5050 | 8052 | http://localhost:8052 | 内部服务 |
| Document Server | 80 | 8085 | http://localhost:8085 | 文档编辑（内部） |

**注意**：用户只需访问 http://localhost:8092 即可使用DocSpace全部功能。其他端口为内部服务或管理用途。
