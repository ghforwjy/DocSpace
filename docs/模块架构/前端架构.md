# 前端服务架构文档

## 1. Node.js 微服务概述

OnlyOffice DocSpace 的前端服务由多个 Node.js 微服务组成，运行在 `onlyoffice-node-services` 容器中，由 Supervisor 进行进程管理。基于 Next.js 框架，支持 SSR（服务端渲染）和 CSR（客户端渲染）。

### 1.1 微服务列表

| 服务名称 | 程序 | 目录 | 用途 |
|----------|------|------|------|
| ASC.Login | server.js | /var/www/products/ASC.Login/login | 用户登录认证 |
| ASC.Editors | server.js | /var/www/products/ASC.Editors/editor | 文档编辑器 |
| ASC.Management | server.js | /var/www/products/ASC.Management/management | 管理后台 |
| ASC.Sdk | server.js | /var/www/products/ASC.Sdk/sdk | SDK服务 |
| ASC.Socket.IO | server.js | /var/www/services/ASC.Socket.IO | WebSocket实时通信 |
| ASC.SsoAuth | app.js | /var/www/services/ASC.SsoAuth | SSO认证 |

## 2. 渲染架构

### 2.1 SSR vs CSR

DocSpace 前端采用混合渲染模式：

| 渲染模式 | 缩写 | 说明 | 适用场景 |
|----------|------|------|----------|
| Server-Side Rendering | SSR | Node.js 服务端执行 React 组件渲染 | 登录页、首页等需要 SEO 的页面 |
| Client-Side Rendering | CSR | 浏览器端执行 JavaScript 渲染 | 管理后台、文档编辑器等 |

### 2.2 关键源码文件

| 文件路径 | 作用 |
|----------|------|
| `client/packages/login/src/app/layout.tsx` | SSR 布局组件，获取 settings |
| `client/packages/login/src/app/(root)/page.tsx` | 登录页面主组件 |
| `client/packages/login/src/utils/actions.ts` | API 调用函数（getSettings, getConfig 等） |
| `client/packages/shared/utils/next-ssr-helper.ts` | SSR 请求构造（createRequest, getAPIUrl） |
| `client/packages/shared/utils/axiosClient.ts` | axios 客户端初始化（initCSR） |
| `client/public/scripts/config.json` | 静态配置文件 |
| `client/packages/login/src/components/Scripts.tsx` | CSR 客户端配置加载 |

### 2.3 SSR 请求流程

```
浏览器请求 /login
       ↓
┌─────────────────────────────────────────────────┐
│  Router (Nginx 8092)                           │
│    设置 x-forwarded-host: localhost:8092        │
│    设置 x-forwarded-proto: http                │
│    转发请求到 onlyoffice-node-services:5011   │
└─────────────────────────────────────────────────┘
       ↓
┌─────────────────────────────────────────────────┐
│  Node.js (Next.js SSR)                         │
│    ↓                                            │
│  layout.tsx 执行 getSettings()                  │
│    ↓                                            │
│  getAPIUrl() 构造 API URL                       │
│    - 优先使用 API_HOST 环境变量                  │
│    - 否则使用 x-forwarded-host header           │
│    ↓                                            │
│  fetch(/api/2.0/settings?withPassword=true)     │
│    ↓                                            │
│  返回 HTML 给浏览器                            │
└─────────────────────────────────────────────────┘
```

## 3. API 请求构造

### 3.1 两套独立的 API 请求体系

DocSpace 前端存在 **两套独立的 API 请求路径**，分别用于 SSR 和 CSR：

| 阶段 | 发起者 | 请求库 | 文件 |
|------|--------|--------|------|
| **SSR** | Node.js 服务端 | Node.js `fetch` | `next-ssr-helper.ts` |
| **CSR** | 浏览器 | axios | `axiosClient.ts` |

### 3.2 SSR 请求构造（服务端渲染）

**文件**: `client/packages/shared/utils/next-ssr-helper.ts`

```typescript
export const getAPIUrl = async (apiSystem?: boolean) => {
  const baseUrl = process.env.API_HOST?.trim() ?? (await getBaseUrl());
  //                              ↑ 第一选择              ↑ fallback
  return `${baseUrl}/${!apiSystem ? API_PREFIX : APISYSTEM_PREFIX}`;
};
```

**SSR 请求地址 fallback 链**：

```
┌─────────────────────────────────────────────────────────────┐
│                   getAPIUrl() 构造地址                       │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  ① process.env.API_HOST                                     │
│     ├─ 值: "http://onlyoffice-dotnet-services:5000"         │
│     ├─ 来源: Supervisor 环境变量配置                         │
│     └─ 状态: 容器内有效 ✅                                   │
│        ↓ 如果未设置                                          │
│  ② getBaseUrl() → x-forwarded-host                         │
│     ├─ 值: "localhost:8092" (来自 Nginx)                    │
│     └─ 状态: 容器内无效 ❌ (localhost 指向容器自己)           │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

**典型问题**：
- `API_HOST` 未设置时，`getBaseUrl()` 返回 `http://localhost:8092`
- 容器内 `localhost:8092` 无法连接 → `fetch failed` → **登录页面空白**

### 3.3 CSR axiosClient 初始化（客户端渲染）

**文件**: `client/packages/shared/utils/axiosClient.ts`

```typescript
initCSR = () => {
  const origin =
    window.ClientConfig?.api?.origin ||   // ① 运行时配置
    apiOrigin ||                          // ② 编译时嵌入
    window.location.origin;               // ③ 浏览器域名
  //                         ↑ 依次尝试，直到找到非空值
};
```

**CSR axios baseURL fallback 链**：

```
┌─────────────────────────────────────────────────────────────┐
│                   initCSR() 构造 baseURL                     │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  ① window.ClientConfig.api.origin                          │
│     ├─ 来源: fetch("/static/scripts/config.json")           │
│     ├─ 当前值: "" (空字符串)                                 │
│     └─ 状态: falsy，继续 fallback                           │
│        ↓                                                    │
│  ② apiOrigin                                               │
│     ├─ 来源: 编译时嵌入的 defaultConfig                      │
│     ├─ 当前值: "http://localhost:8092"                     │
│     └─ 状态: 有值 ✅                                        │
│        ↓ 如果 apiOrigin 也是空                              │
│  ③ window.location.origin                                  │
│     ├─ 来源: 浏览器地址栏                                    │
│     └─ 状态: 最终 fallback                                  │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 3.4 两套体系的对比与关联

| 维度 | SSR | CSR |
|------|-----|-----|
| **发起位置** | Node.js 容器内 | 浏览器 |
| **请求库** | Node.js `fetch` | axios |
| **第一选择** | `API_HOST` 环境变量 | `ClientConfig.api.origin` |
| **第二选择** | `x-forwarded-host` header | `apiOrigin` (编译时嵌入) |
| **最终 fallback** | N/A | `window.location.origin` |
| **失败表现** | 页面空白（无登录控件） | API 请求失败（页面已渲染） |
| **容器内可用性** | ❌ 需要 `API_HOST` | ✅ 浏览器可访问 |

### 3.5 当前环境实际值

| 变量 | 实际值 | 状态 |
|------|--------|------|
| `API_HOST` | 未设置 | ❌ SSR 会失败 |
| `x-forwarded-host` | `localhost:8092` | ❌ 容器内无效 |
| `ClientConfig.api.origin` | `""` (空) | falsy，继续 |
| `apiOrigin` | `http://localhost:8092` | ✅ CSR 可用 |

**结论**：CSR 能工作（因为 `apiOrigin` 有值），SSR 会失败（因为 `API_HOST` 未设置）。

### 3.5 SSR vs CSR 请求构造对比

| 特性 | SSR | CSR |
|------|-----|-----|
| 发起时机 | 服务端渲染时 | 客户端 hydrate 后 |
| 请求函数 | `getSettings()` in actions.ts | `axiosClient.initCSR()` |
| 请求库 | Node.js fetch | axios |
| baseURL 来源 | `API_HOST` 环境变量 或 `x-forwarded-host` | `ClientConfig.api.origin` 或 `window.location.origin` |
| 超时控制 | 无 | `apiTimeout` |
| 失败表现 | 页面空白（无登录控件） | API 请求失败（但页面已渲染） |

## 8. 已知问题

### 8.1 SSR 时 API 请求失败（API_HOST 缺失）

**问题描述**：SSR（服务端渲染）时，Node.js 内部调用 API 失败，导致登录页面显示空白。

**现象**：
- 访问 `/login` 时页面显示空白背景，没有登录控件
- 浏览器控制台显示 `wrongPortalNameUrl: ""`
- Node.js 容器日志显示 `Error in getSettings: TypeError: fetch failed`

**根本原因**：

1. SSR 时 Node.js 内部调用 `getSettings()` API
2. 代码使用 `getBaseUrl()` 获取 `x-forwarded-host` = `localhost:8092`
3. 但容器内 `localhost` 指向**容器自己**，不是 Router
4. 导致 `fetch` 失败 → 登录控件不显示

**证据**：

```bash
# 容器内测试 - 失败
docker exec onlyoffice-node-services curl -s -o /dev/null -w '%{http_code}' http://localhost:8092/api/2.0/settings
# 返回: 000 (连接失败)

# 容器内用服务名测试 - 成功
docker exec onlyoffice-node-services curl -s -o /dev/null -w '%{http_code}' http://onlyoffice-dotnet-services:5000/api/2.0/settings
# 返回: 200
```

**影响**：
- `/login/*` → 空白页面（无登录控件）
- SSR 时所有 API 调用失败（getSettings, getColorTheme, getThirdPartyProviders 等）

**修复方案**：

在 Supervisor 配置中添加 `API_HOST` 环境变量：

```ini
environment=SERVICE_LOGIN_PORT="5011",...,API_HOST="http://onlyoffice-dotnet-services:5000"
```

**⚠️ 必须执行 `reread` + `update`**：

单独 `restart` 不会重新加载环境变量，必须执行：

```bash
# 重新读取配置
docker exec onlyoffice-node-services supervisorctl reread

# 应用配置变更（会重启相关服务）
docker exec onlyoffice-node-services supervisorctl update

# 验证服务状态
docker exec onlyoffice-node-services supervisorctl status ASC.Login
```

**验证修复**：

```bash
# 测试容器内 API 访问
docker exec onlyoffice-node-services curl -s -o /dev/null -w '%{http_code}' http://onlyoffice-dotnet-services:5000/api/2.0/settings
# 返回 200 表示成功
```

### 8.2 wrongPortalNameUrl 为空字符串

**现象**：
- 浏览器控制台显示 `wrongPortalNameUrl: ""`（空字符串）

**原因**：
- `public/scripts/config.json` 中 `wrongPortalNameUrl` 配置为空
- 不影响登录控件显示，但影响 portal-not-found 时的重定向行为
