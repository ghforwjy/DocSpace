# 登录服务调用流程

## 1. 整体架构

DocSpace 系统登录涉及以下组件：

| 组件 | 容器 | 端口 | 技术栈 | 说明 |
|------|------|------|--------|------|
| Router | onlyoffice-router | 8092 | Nginx | 反向代理入口 |
| ASC.Login | onlyoffice-node-services | 5011 | Node.js/Next.js | 登录页面和认证API |
| ASC.Web.Api | onlyoffice-dotnet-services | 5000 | .NET | 主API服务 |
| ASC.Identity | onlyoffice-java-services | 8080 | Java/Spring | 身份授权服务 |
| ASC.People | onlyoffice-dotnet-services | 5004 | .NET | 用户管理服务 |
| API Cache | onlyoffice-java-services | 5100 | Java/Spring | API缓存服务 |

## 3. 详细流程步骤

### 步骤1：用户访问入口
```
URL: http://domain:8092/
```
用户浏览器访问 DocSpace 入口地址。

### 步骤2：Router返回静态资源
Router (Nginx) 根据配置返回前端静态文件（客户端应用）。

### 步骤3：访问登录页面
```
URL: http://domain:8092/login
```
用户点击登录按钮或访问登录页面。

### 步骤4：Router代理到Login服务
Router 根据 upstream.conf 配置将请求代理到 Node.js 容器：

```nginx
# upstream.conf
map onlyoffice-node-services:5011 $service_login

# onlyoffice.conf
location /login {
    proxy_pass http://$service_login;
}
```

**目标地址**: `onlyoffice-node-services:5011`

### 步骤5：Login服务返回页面
ASC.Login 服务（Next.js）返回登录页面 HTML。

### 步骤6：用户提交登录表单
用户在登录页面输入用户名/密码，提交到：
```
POST http://domain:8092/api/2.0/authentication/
```

### 步骤7：Router代理请求
这里存在**路由配置问题**（详见问题2）

**正确流程应该是**：
```nginx
# /api/2.0/authentication/* 应该路由到 Node.js Login服务
location ~* /authentication {
    proxy_pass http://$service_login;  # 5011
}
```

**但当前配置**：
```nginx
location /api/2.0 {
    proxy_pass http://$service_api;  # 被路由到 5000 .NET服务
}
```

### 步骤8：身份验证流程
1. ASC.Web.Api (5000) 接收登录请求 → **错误：应该在5011**
2. 调用 ASC.Identity.Authorization (8080) 进行身份验证
3. ASC.Identity.Authorization 调用 ASC.People (5004) 获取用户信息
4. 验证用户凭证

### 步骤12：返回登录结果
验证成功后，返回 token/session，用户登录成功。

### 5.3 SSR API请求构造

**文件**: `client/packages/shared/utils/next-ssr-helper.ts`

```typescript
export const getAPIUrl = async (apiSystem?: boolean) => {
  const baseUrl = process.env.API_HOST?.trim() ?? (await getBaseUrl());
  const baseAPIUrl = `${baseUrl}/${!apiSystem ? API_PREFIX : APISYSTEM_PREFIX}`;
  return baseAPIUrl;
};

export const getBaseUrl = async () => {
  const hdrs = await headers();
  const host = hdrs.get("x-forwarded-host");
  const proto = hdrs.get("x-forwarded-proto");
  return `${proto}://${host}`;
};
```

**API请求URL构造逻辑**：
1. 如果设置了 `API_HOST` 环境变量 → 使用它
2. 否则使用 `x-forwarded-host` header（来自原始请求）

### 5.6 API 请求地址 fallback 链

DocSpace 前端存在 **两套独立的 API 请求路径**，分别用于 SSR 和 CSR：

| 阶段 | 发起者 | 请求库 | 文件 |
|------|--------|--------|------|
| **SSR** | Node.js 服务端 | Node.js `fetch` | `next-ssr-helper.ts` |
| **CSR** | 浏览器 | axios | `axiosClient.ts` |

#### SSR 请求地址 fallback 链

```
┌─────────────────────────────────────────────────────────────┐
│                   getAPIUrl() 构造地址                       │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  ① process.env.API_HOST                                     │
│     ├─ 值: "http://onlyoffice-dotnet-services:5000"         │
│     ├─ 来源: Supervisor 环境变量配置                         │
│     └─ 状态: 容器内有效 ✅                                   │
│        ↓ 如果未设置                                          │
│  ② getBaseUrl() → x-forwarded-host                         │
│     ├─ 值: "localhost:8092" (来自 Nginx)                    │
│     └─ 状态: 容器内无效 ❌ (localhost 指向容器自己)           │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

#### CSR axios baseURL fallback 链

```
┌─────────────────────────────────────────────────────────────┐
│                   initCSR() 构造 baseURL                     │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  ① window.ClientConfig.api.origin                          │
│     ├─ 来源: fetch("/static/scripts/config.json")           │
│     ├─ 当前值: "" (空字符串)                                 │
│     └─ 状态: falsy，继续 fallback                           │
│        ↓                                                    │
│  ② apiOrigin                                               │
│     ├─ 来源: 编译时嵌入的 defaultConfig                      │
│     ├─ 当前值: "http://localhost:8092"                     │
│     └─ 状态: 有值 ✅                                        │
│        ↓ 如果 apiOrigin 也是空                              │
│  ③ window.location.origin                                  │
│     ├─ 来源: 浏览器地址栏                                    │
│     └─ 状态: 最终 fallback                                  │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

#### 两套体系的对比与关联

| 维度 | SSR | CSR |
|------|-----|-----|
| **发起位置** | Node.js 容器内 | 浏览器 |
| **请求库** | Node.js `fetch` | axios |
| **第一选择** | `API_HOST` 环境变量 | `ClientConfig.api.origin` |
| **第二选择** | `x-forwarded-host` header | `apiOrigin` (编译时嵌入) |
| **最终 fallback** | N/A | `window.location.origin` |
| **失败表现** | 页面空白（无登录控件） | API 请求失败（页面已渲染） |
| **容器内可用性** | ❌ 需要 `API_HOST` | ✅ 浏览器可访问 |

#### 当前环境实际值

| 变量 | 实际值 | 状态 |
|------|--------|------|
| `API_HOST` | 未设置 | ❌ SSR 会失败 |
| `x-forwarded-host` | `localhost:8092` | ❌ 容器内无效 |
| `ClientConfig.api.origin` | `""` (空) | falsy，继续 |
| `apiOrigin` | `http://localhost:8092` | ✅ CSR 可用 |

**结论**：CSR 能工作（因为 `apiOrigin` 有值），SSR 会失败（因为 `API_HOST` 未设置）。

## 6. 已知问题

### 问题1：SSR时API请求失败导致登录控件不显示

**现象**:
- 访问 `/login` 时页面显示空白背景，没有登录控件
- 浏览器控制台显示 `wrongPortalNameUrl: ""`
- Node.js容器日志显示 `Error in getSettings: TypeError: fetch failed`

**排查结果**:
```bash
# 容器内测试 - 失败
docker exec onlyoffice-node-services curl -s -o /dev/null -w '%{http_code}' http://localhost:8092/api/2.0/settings
# 返回: 000 (连接失败)

# 容器内用服务名测试 - 成功
docker exec onlyoffice-node-services curl -s -o /dev/null -w '%{http_code}' http://onlyoffice-router:8092/api/2.0/settings
# 返回: 200

docker exec onlyoffice-node-services curl -s -o /dev/null -w '%{http_code}' http://onlyoffice-dotnet-services:5000/api/2.0/settings
# 返回: 200
```

**根本原因**:
1. SSR时 Node.js 内部调用 `getSettings()` API
2. 代码使用 `getBaseUrl()` 获取 `x-forwarded-host` = `localhost:8092`
3. 但容器内 `localhost` 指向**容器自己**，不是Router
4. 导致 `fetch` 失败 → `settings` 返回 `undefined` 或抛出异常
5. `page.tsx` 中 `settings` 不是对象且不是字符串 `"portal-not-found"`，返回 `null` → **空白页面**

**证据**:
- 配置文件: `/etc/supervisor/conf.d/supervisord.conf`
- 当前环境变量: `SERVICE_LOGIN_PORT="5011"` ✅ 已设置
- **缺少**: `API_HOST` 环境变量 ❌

**影响**:
- `/login/*` → 空白页面（无登录控件）
- SSR时所有API调用失败（getSettings, getColorTheme, getThirdPartyProviders等）

### 问题2：wrongPortalNameUrl 为空字符串

**现象**:
- 浏览器控制台显示 `wrongPortalNameUrl: ""`（空字符串）

**原因**:
- `public/scripts/config.json` 中 `wrongPortalNameUrl` 配置为空
- 不影响登录控件显示，但影响 portal-not-found 时的重定向行为

**建议修复**:
如需自定义错误页面，修改 `public/scripts/config.json` 中的 `wrongPortalNameUrl` 值

### initCSR 输出参数

| 参数 | 含义 | 诊断价值 |
|------|------|----------|
| `defaultConfig` | 编译时嵌入的配置 | 对比运行时配置是否一致 |
| `apxiosConfig.baseURL` | axios 请求基础URL | ⚠️ **如果与预期不符，说明 ClientConfig 或 defaultConfig 有问题** |
| `apxiosConfig.timeout` | axios 超时 | 确认是否使用配置值 |
| `ClientConfig` | 运行时加载的配置 | 与 `defaultConfig` 对比 |
| `paymentsURL` | 支付页面URL | 确认构造是否正确 |

### 诊断流程

当登录页面出现问题时，按以下顺序检查：

```
1. 检查控制台是否有输出
   ├── 有 config 输出 → CSR 脚本执行正常
   ├── 无 config 输出 → fetch config.json 失败，检查 Router 日志
   │
2. 检查 initCSR 输出
   ├── apxiosConfig.baseURL 是否正确
   ├── ClientConfig 与 defaultConfig 是否一致
   │
3. 检查 API 请求（浏览器 Network 面板）
   ├── /api/2.0/settings 是否返回 200
   ├── 响应内容是否是有效的 settings 对象
   │
4. 如果 settings 返回错误
   ├── 404 → portal-not-found，检查 wrongPortalNameUrl
   ├── 403 → access-restricted，权限问题
   └── 其他 → 检查 .NET Services 是否正常运行
```

## 7. 修复方案

### 问题1修复：添加 API_HOST 环境变量

**问题**：SSR 时 API 请求失败，导致登录控件不显示

**修复步骤**：

#### 1. 修改 Supervisor 配置文件

在容器内 `/etc/supervisor/conf.d/supervisord.conf` 的 `[supervisord]` 段添加：

```ini
environment=SERVICE_LOGIN_PORT="5011",...,API_HOST="http://onlyoffice-dotnet-services:5000"
```

#### 2. 重新加载配置（关键步骤）

⚠️ **必须执行 `reread` + `update`**，单独 `restart` 不会重新加载环境变量：

```bash
# 重新读取配置
docker exec onlyoffice-node-services supervisorctl reread

# 应用配置变更（会重启相关服务）
docker exec onlyoffice-node-services supervisorctl update

# 验证服务状态
docker exec onlyoffice-node-services supervisorctl status ASC.Login
```

#### 3. 验证修复

```bash
# 测试容器内 API 访问
docker exec onlyoffice-node-services curl -s -o /dev/null -w '%{http_code}' http://onlyoffice-dotnet-services:5000/api/2.0/settings
# 返回 200 表示成功

# 测试登录页面
curl -s -o /dev/null -w '%{http_code}' http://localhost:8092/login
```

#### ⚠️ 常见错误

| 错误操作 | 结果 |
|----------|------|
| 只执行 `supervisorctl restart ASC.Login` | 配置未重新加载，环境变量不生效 |
| 只执行 `supervisorctl reread` | 配置已读取但未应用 |
| 正确操作 `reread` + `update` | 配置正确加载，服务正常启动 |

具体修改位置：`/etc/supervisor/conf.d/supervisord.conf` (Node.js容器内)
