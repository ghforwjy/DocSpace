# 登录服务调用流程

## 1. 整体架构

DocSpace 系统登录涉及以下组件：

| 组件 | 容器 | 端口 | 技术栈 | 说明 |
|------|------|------|--------|------|
| Router | onlyoffice-router | 8092 | Nginx | 反向代理入口 |
| ASC.Login | onlyoffice-node-services | 5011 | Node.js/Next.js | 登录页面和认证API |
| ASC.Web.Api | onlyoffice-dotnet-services | 5000 | .NET | 主API服务 |
| ASC.Identity | onlyoffice-java-services | 8080 | Java/Spring | 身份授权服务 |
| ASC.People | onlyoffice-dotnet-services | 5004 | .NET | 用户管理服务 |
| API Cache | onlyoffice-java-services | 5100 | Java/Spring | API缓存服务 |

## 3. 详细流程步骤

### 步骤1：用户访问入口
```
URL: http://domain:8092/
```
用户浏览器访问 DocSpace 入口地址。

### 步骤2：Router返回静态资源
Router (Nginx) 根据配置返回前端静态文件（客户端应用）。

### 步骤3：访问登录页面
```
URL: http://domain:8092/login
```
用户点击登录按钮或访问登录页面。

### 步骤4：Router代理到Login服务
Router 根据 upstream.conf 配置将请求代理到 Node.js 容器：

```nginx
# upstream.conf
map onlyoffice-node-services:5011 $service_login

# onlyoffice.conf
location /login {
    proxy_pass http://$service_login;
}
```

**目标地址**: `onlyoffice-node-services:5011`

### 步骤5：Login服务返回页面
ASC.Login 服务（Next.js）返回登录页面 HTML。

### 步骤6：用户提交登录表单
用户在登录页面输入用户名/密码，提交到：
```
POST http://domain:8092/api/2.0/authentication/
```

### 步骤7：Router代理请求
这里存在**路由配置问题**（详见问题2）

**正确流程应该是**：
```nginx
# /api/2.0/authentication/* 应该路由到 Node.js Login服务
location ~* /authentication {
    proxy_pass http://$service_login;  # 5011
}
```

**但当前配置**：
```nginx
location /api/2.0 {
    proxy_pass http://$service_api;  # 被路由到 5000 .NET服务
}
```

### 步骤8：身份验证流程
1. ASC.Web.Api (5000) 接收登录请求 → **错误：应该在5011**
2. 调用 ASC.Identity.Authorization (8080) 进行身份验证
3. ASC.Identity.Authorization 调用 ASC.People (5004) 获取用户信息
4. 验证用户凭证

### 步骤12：返回登录结果
验证成功后，返回 token/session，用户登录成功。

## 4. 关键路由配置

### Router当前路由规则（存在问题）

| 路径 | 配置目标 | 实际目标 | 状态 |
|------|----------|----------|------|
| `/login/*` | ASC.Login | 5011 Node.js | ⚠️ 502错误 |
| `/api/2.0/authentication/*` | ASC.Login | 5011 Node.js | ❌ 被路由到5000 |
| `/api/2.0/*` | ASC.Web.Api | 5000 .NET | ✅ 正常 |
| `/people/*` | ASC.People | 5004 .NET | ✅ 正常 |

## 5. 已知问题

### 问题1：Node.js服务监听地址问题

**现象**: 访问 `/login` 时返回 502 Bad Gateway 或 504 Gateway Timeout

**原因**: ASC.Login 服务启动时只监听容器内部IP，未监听 `0.0.0.0`

**证据**:
```
Starting server with environment: NODE_ENV=production, PORT=5011
   ▲ Next.js 16.0.10
   - Local:         http://b522090c7402:5011
   - Network:       http://b522090c7402:5011
```

服务只绑定了容器内部 IP，未绑定 `0.0.0.0` 或 `127.0.0.1`。

**影响范围**:
- `/login/*` → ASC.Login (5011)
- `/doceditor/*` → ASC.Editors (5013)
- `/management/*` → ASC.Management (5015)
- `/socket.io/*` → ASC.Socket.IO (9899)
- `/sso/*` → ASC.SsoAuth (9834)

### 问题2：认证API路由错误

**现象**: 
- `GET /api/2.0/authentication/status` 返回 `405 Method Not Allowed`
- `POST /api/2.0/authentication/` 返回 `401 Unauthorized` (被错误路由到.NET服务)

**原因**: `/api/2.0/authentication/*` 路径被错误路由到 .NET 的 ASC.Web.Api (5000)，而不是 Node.js 的 ASC.Login (5011)

**当前错误配置**:
```nginx
location /api/2.0 {
    proxy_pass http://$service_api;  # 匹配所有 /api/2.0/* 路径，包括 /authentication
}
```

**应该添加的配置**:
```nginx
location ~* /authentication {
    proxy_pass http://$service_login;  # 5011
}
```

**影响**: 认证API无法正常工作，用户登录失败

### 问题3：认证流程中断

由于问题1和2，用户访问登录页面时：
1. Router 无法连接到 ASC.Login 服务 → 502错误
2. 即使连上了，认证API也被路由到错误的服务 → 405/401错误
3. 用户无法看到登录页面
4. 整个登录流程无法完成

## 7. 修复方案

### 问题1修复：Node.js服务监听地址

需要修改 ASC.Login 服务的启动命令，添加监听地址参数：

```bash
# 添加 --app.host=0.0.0.0 或环境变量 HOST=0.0.0.0
/usr/local/bin/node server.js --app.port=5011 --app.host=0.0.0.0 ...
```

具体修改位置：`/etc/supervisor/conf.d/supervisord.conf` (Node.js容器内)

### 问题2修复：认证API路由配置

需要在 Router 的 Nginx 配置中添加 authentication 路由：

```nginx
# 在 location /api/2.0 之前添加
location ~* /authentication {
    proxy_pass http://$service_login;
}
```

具体修改位置：`/etc/nginx/conf.d/onlyoffice.conf` (Router容器内)
